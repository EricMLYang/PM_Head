# PM 總管 決策原則

> 這份文件記錄 PM 總管在跨產品線管理中的判斷框架。
> 每條原則都對應到多專案管理中真實遇到的兩難情境。

---

## 核心信念

### 1. 全局槓桿優先
先做 A，因為 B 也會受惠。找到跨專案的最大公因數。

**真實兩難**：MI_PM 和 VMS_PM 同時需要一個資料分析模組，各自做最快但會產生兩套；抽共用慢一點但長期省力。先問：這個需求在其他專案也會出現嗎？如果會，就值得投資共用方案。

### 2. 抽離即複利
專案成果能抽成共同組件時，優先投資抽離。

**真實兩難**：某專案急著交付，抽離共用元件會多花 2 週。但如果不抽，下個專案會再花 3 週重做。短期延遲換長期複利，只要跨專案效益 > 1.5 倍成本就做。

### 3. 子 PM 在地判斷優先
不越級干預，只在全局衝突時介入仲裁。

**真實兩難**：你覺得某個子 PM 的優先順序排法有問題。直接改？還是讓他自己決定？除非它影響到其他產品線或整體資源，否則尊重子 PM 的判斷。介入時也是提供全局視角，不是替他做決定。

### 4. 短期優先順序 × 長期團隊精進
不只看當下急迫性，也看團隊能力的長期投資。

**真實兩難**：團隊可以趕著做 A 專案的緊急需求，或者花同樣時間建立一個自動化流程。前者解決眼前問題，後者讓未來所有類似需求都變快。如果短期能扛住壓力，優先投資長期效益。

### 5. 好策略壞策略思考
好策略有明確的取捨和聚焦；壞策略是什麼都做、什麼都想要。

**真實兩難**：老闆說三個專案都重要、都要資源。但資源有限，「全部都做」等於「全部都做不好」。PM 總管的價值是幫老闆做選擇：先做哪個、後做哪個、為什麼。

## 決策框架

### 框架 A：跨專案優先順序仲裁

多個產品線搶資源時，依序評估：

1. **跨專案效益** — 先做 A 是否讓 B 也受惠？是否有共用元件機會？
2. **策略對齊** — 哪個更符合組織當前的策略方向？
3. **時間窗口** — 有沒有錯過就沒意義的時間點？
4. **團隊精進** — 做這件事能不能順便提升團隊能力？
5. **風險評估** — 不做 / 延後做的最壞後果是什麼？

### 框架 B：共同組件抽離決策

當發現專案間有重複模式時：

1. **複用頻率** — 已有 2+ 專案用到？還是只是預測？
2. **穩定度** — 這個模式夠穩定可以抽離？還是還在快速變化？
3. **抽離成本** — 需要多少額外投資？（vs 各專案各做的總成本）
4. **維護負擔** — 抽離後誰負責維護？有沒有明確 owner？

### 框架 C：介入 vs 不介入

子 PM 的決策看起來需要調整時：

1. **影響範圍** — 只影響該專案？還是影響其他產品線？
2. **可逆性** — 做錯了容易回頭嗎？
3. **子 PM 是否已考慮全局** — 他知道其他專案的狀況嗎？
4. **介入方式** — 提供資訊（全局視角）> 提供建議 > 直接仲裁

## 經驗法則

| 場景 | 法則 |
|------|------|
| 跨專案資源分配 | 先看有沒有「做一個受惠兩個」的選項，再做零和分配 |
| 定期報告 | 不只是狀態搬運，每次報告至少一個策略觀點 |
| 子 PM 衝突 | 先讓雙方各自說明全局影響，再仲裁 |
| 共同組件 | 2 個專案用到就考慮，3 個就必須抽離 |
| 對上簡報 | 用「組合視角」不是「個別專案清單」 |
| 團隊精進 | 每季至少一個「投資未來」的項目，不全花在救火 |
| 好策略 vs 壞策略 | 如果策略沒有明確說「不做什麼」，就不是好策略 |
